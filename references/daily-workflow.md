# 每日综合工作流

仅在 `daily` 工作流中读取本文件。按需组装报告，不得默认加载所有组件
reference。

## 仅加载所需内容

1. 始终读取 `decision-integrity-rules.md`、`market-workflow.md` 和
   `behavioral-workflow.md`，并收集其决策完整性、大盘与群体行为清单。
2. 仅当用户提供一只或多只自选股时读取 `stock-workflow.md`。
3. 运行上升初段候选发现时，仅阅读 `screening-rules.md` 的
   **上升初段候选**部分。
4. 仅当用户指定板块时，阅读 `screening-rules.md` 的
   **低位活跃龙头候选**部分。
5. 生成 HTML 前读取 `report-contract.md`；存在 T-1 报告时，核心 Skill
   已要求读取 `prior-report-rules.md`。

## 执行顺序

1. 解析已完成交易日并校验输入。
2. 收集大盘数据并确定数据质量。
3. 形成市场层行为结论；缺少时序或生态数据时按证据充分度降级。
4. 读取 `prior-report-rules.md`，随后在可用时结构化复盘匹配的 T-1 `daily`
   context。
5. 仅对用户提供的自选股运行个股分析和个股行为分析。
6. 仅当用户明确请求上升初段候选时，在沪深主板、创业板的可验证股票池中运行
   候选发现，并披露股票池来源和实际覆盖范围；否则将
   该区块标为“未运行：用户未请求上升初段候选”。
7. 仅对用户指定板块中的沪深主板、创业板成员运行低位活跃龙头筛选；未指定时
   将该区块标为
   “未运行：未指定板块”。
8. 固定筛选和排序完成后，只对最终候选补充行为分析，不得据此替换候选或改变
   名次。
9. 对请求个股和最终候选检查可交易性，并在 T-1 计划中固定相对评价基准。
10. 生成一份报告，明确筛选覆盖、所有跳过区块和数据质量限制。

先依据筛选规则和收集到的证据确定候选资格，再撰写大盘/个股叙述。不得因
前文的定性结论增删候选；市场状态只能用于说明候选的交易环境。
